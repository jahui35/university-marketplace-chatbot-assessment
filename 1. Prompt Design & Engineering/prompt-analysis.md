# Prompt Design Analysis: UniMarketplace Assistant

## Structural and Formatting Decisions

I chose XML-based hierarchical formatting over plain markdown or JSON for the system prompt because it provides superior separation of concerns while remaining both human and machine-readable. XML's tag-based structure allows clear compartmentalization of role definition, task specifications, conditional logic, and safety protocols—making the prompt easier to maintain as university policies evolve, such as when new prohibited items are added or campus regulations change. The nested hierarchy also aids model parsing which embeds the reasoning framework within core capabilities signals that safety checks and intent classification are foundational to all operations. Attributes like id="unverified_user_post" provide metadata that could support automated testing or version-control diff tracking. For a marketplace chatbot where requirements change frequently, this structure reduces the risk of unintended side effects when updating individual components.

## User Scenarios and Safety Architecture

The prompt explicitly addresses four distinct user personas through the user_scenarios section: buyers searching and evaluating items, sellers creating and managing listings, new users requiring account verification, and users reporting safety concerns. Each persona has dedicated intent mappings that guide the chatbot's response strategy. For safety guardrails, I implemented a multi-layer approach where prohibited_items lists specific categories like vapes, alcohol, and counterfeit goods that trigger automatic blocking; detection_mechanism specifies AI scanning with a 95% confidence threshold before human review; and escalation_protocol defines trigger keywords (fraud, harassment, scam) that immediately generates ticket IDs and hand off conversation transcripts to moderators. The layered safety architecture ensures policy violations are caught at multiple points, during listing creation, in conversation, and through user reports, while maintaining clear escalation pathways that protect both users and the institution from liability.

## Structured Output and University Context Integration

For structured outputs, the prompt uses JSON schemas triggered by specific actions like step by step instructions, search results, and escalation tickets. This enables downstream automation for testing and allows the frontend to render responses consistently across mobile and web interfaces. The default response style remains natural language for conversational queries, balancing flexibility with structure. University specific context is embedded throughout: academic_calendar notes peak furniture sales during end-of-semester hall clearance and low activity during exam weeks, allowing the chatbot to anticipate seasonal demand spikes; campus_policies specify dorm storage restrictions and pickup hours (9am-8pm) to set realistic user expectations; safe_meetup_zones list concrete locations like LT15 and Koufu entrance rather than generic advice; and verification requirements mandate .edu.sg email validation against the university database before enabling posting privileges. Moreover, the prompt makes sure that academic policies of the university are followed to prevent any sale or distribution of copyrighted materials, and prevents academic dishonesty. These contextual details transform a generic marketplace chatbot into one that understands the university's unique community dynamics, building trust through demonstrated institutional knowledge while enforcing policies that align with student life realities.
